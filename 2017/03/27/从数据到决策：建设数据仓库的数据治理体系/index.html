从数据到决策：建设数据仓库的数据治理体系

数据治理是对企业数据资源进行管控治理的组织结构策略和方法论。它将企业中跨部门、跨功能的团队联合起来，组成解决数据资源问题的统一战线，构建了一切信息流程的权责体系，使业务和IT共同解决关键的数据问题。

[TOC]

概述

定义

数据治理 - DGI (Data Governance Institute)
“数据治理/管控是一个管理一切与信息有关的流程的权责体系；它的执行需按照一个经各方认可的、陈述何方在何种情形下、能以何种方式使用何种信息、进行何种行动的机制来进行。”

数据治理的必要性

企业高效运营的需要

  • 数据治理通过高质量的数据改善决策能力
  • 数据治理有助于企业发现商业机会
  • 数据治理有助于降低IT建设及运维成本
  • 高质量的数据对于企业业务系统的安全稳定运营极其关键

企业风险管理的需要

  • 风险控制的应用受制于信息资源的规模和质量
  • 资产分类的准确性有助于降低风险

业务管理创新需要

  • 有助于业务流程的改进，资源的优化，提高业务及管控能力
  • 高质量的数据是实现商业只能的基础，准确的分析数据有助于做出正确的业务及管理创新

数据管控的问题

企业发展到一定阶段时，数据管控方面普遍容易存在以下问题：

  1. 数据多样化，缺少统一标准
  2. 源数据变化造成数据平台数据混乱
  3. 统计结果不准确、数据指标异常波动
  4. 隐蔽性的数据质量问题，数据平台层次复杂、不能快速定位原因
  5. 数据质量低下、应用定义不清、报表利用率低
  6. 数据标准缺乏结构化管理，集成困难
  7. 数据分散，形成信息孤岛，共享困难
  8. 已然形成大数据，但针对其应用处理困难
  9. 主数据安全
  10. 缺少灾备、迁移、热切换，不能保障数据平台的高可用性
  11. 缺少敏捷的需求管控机制、取数需求到数据获得流程繁冗，需求方需要的决策时间被延长
  12. 数据生产完成后，需求方获取数据的渠道不够便捷，不能第一时间掌握需要的信息
  13. 出了问题再补漏，数据管理部门和生产部门互相推脱责任

综上，对问题进行分类梳理，不难发现企业在数据管控需要重点处理的问题包括：数据标准（1,2,5,14）、数据质量（3,4,5）、数据集成（6,7,8）、元数据管理（5,6,7,8）、主数据管理（1,2,7,8）、数据安全（9,10）、数据服务（11,12）、权责体系（13）。有些问题只是单个原因导致的，而还有部分问题的出现并不是由单方面的原因导致的，例如报表利用率低的原因可能是数据质量、指标定义不清晰（业务理解存在歧义）、获取数据的渠道不够便捷、业务理解不深入导致数据价值低下等。由此可见，单项的数据管控可以解决一部分问题，但是不能从根本上解决全部问题。

数据治理解决方案

首先，明确数据管理核心目标是：保障核心业务正常运作，提升数据价值，深度挖掘数据应用价值。
然后，从数据生命周期的管控考虑，数据治理将主要集中在以下几个方面：

  • 建立数据标准体系
    建立企业视角的通用数据标准，如客户定义、客户分类、客户信息等。分析梳理业务流程，设计业务主数据指标体系，形成数据标准规范。同时规范化管理构成数据平台的业务和技术基础设施，包括数据管控制度、流程规范文档、信息项定义等。
  • 建立度量的数据质量稽核标准，提升主数据数据质量
    探查、解决、处理数据质量问题，对数据质量问题进行分类、关联绩效、问题来源的业务/技术培训等，提升数据价值。
  • 主数据管理，对数据进行集成共享
    主数据数据模型建设，将各类处理过的优质生产数据统一集中存储，便于调用、共享。支持大数据应用集成。
  • 元数据管理，掌握数据关系脉络，管理知识库
    实现对数据间流转、依赖关系的影响和血缘分析，明确数据关系脉络；知识库存储（定义、分类、信息项及其业务描述、技术定义等）；管理归属（管理信息项的系统、部门、人员）。
  • 数据安全管控
    完备数据平台安全防守，应对来自后端、前端的安全风险，提高数据平台的高可用性。
  • 数据服务集成
    打通需求平台到应用建设的层层审批，缩短从业务到IT的距离；建立数据需求确认-数据开发-数据发布-订阅/推送的流程，以便决策者能对市场变化做出敏捷反应。
  • 权责体系建设
    业务模块和技术模块资源分配，权责到人。如果出现问题，由谁处理，如果未处理，向谁报告，由谁替补处理。

最后就是项目落实。这需要不断跟进项目进展、合理安排资源。这个过程中，沟通是及时完成的保障，执行是项目关键。

  • 制定数据应用规则（数据集成、分发规则）
  • 制定数据模型（统一数据视图）

技术层面

  • 数据探查
  • 数据清洗、质量提升
  • 数据集成、监控
  • 数据安全

执行层面

  • 业务梳理
  • 数据分析
  • 绩效评估
  • 计划-执行-检查-处理 PDCA循环管控机制

数据质量综合评估

可访问性

  • 源数据可访问性
  • 结果数据和报表页面的可访问性
  • 用户体验（查询性能、等待时长、展现格式）

可用性

  • 正确性：唯一性检查、非空检查、类型检查、格式检查、值域检查、代码检查、指标标准偏差检查、数据集市指标准确性检查。
  • 完整性：外键约束检查（参照完整性，指一个表A的外键不包含无效的键值，例如，借据表中记录了合同号，但是在合同表中无法找到相关记录）、源数据与主数据映射关系检查。
  • 一致性：业务规则检查、指标符合钩稽关系、不同粒度维度下指标核算。

及时性
数据平台数据生成的及时性检查

数据质量问题分析

对历史出现过的的数据质量问题进行影响性量化分析，对潜在的数据问题进行预判和防范。可从以下几方面分析：

  • 问题分布分析：收集质量问题的统计分布情况，针对性地进行优化，改善数据质量
  • 问题趋势分析：对数据问题发生地趋势进行监控，做好潜在数据质量问题的防备
  • 稽核执行报告：将问题发现-评估影响-解决过程形成报告机制，提高数据生产者、管理者与消费者的信息传递速度

数据质量问题管理

  • 问题分布
  • 问题处理
  • 问题归档

数据质量知识管理

  • 知识库查询：可通过查询历史数据问题的归档文件回溯该问题的成因、影响、处理过程。

元数据管理

元数据（Metadata）是关于数据的数据，即对数据的描述信息。根据其属性的不同，元数据可分为技术元数据和业务元数据。元数据管理是元数据的定义、收集、管理和发布的方法、工具及流程的集合，通过完成对相关业务元数据及技术元数据的集成及应用，提供数据路径、数据归属信息，并对业务术语、文档进行集中管理，借助变更报告、影响分析以及业务术语管理等应用，以此保证数据的完整性、控制数据质量、减少业务术语歧义和建立业务人员之间、技术人员之间，以及双方的沟通平台。元数据使得用户可以掌握数据的历史情况，如数据从哪里来？流通时间有多长？更新频率是多大？数据元素的含义是什么？对它已经进行了哪些计算、转换和筛选等等。在需求不确定情况下，在瞬间万变的商业环境下，元数据可以更好的支持需求的变化，降低项目风险。元数据管理是企业级数据仓库中的关键组件，贯穿于建立数据仓库的整个过程。

Metadata is structured information that describes the characteristics/attributes of a data element. Things like key identifiers, data descriptions and file types are examples of metadata.（From CWM）

元数据的作用

  • 提供一份数据定义和元素的详细示意图，有利于发现冗余的定义和元素
  • 帮助用户理解数据的来龙去脉、关系和相关属性
  • 数据集成所必须的：参照性、引用性、血缘分析、影响分析、变化分析
  • 减少数据冗余，增加信息共享，避免异构结构定义

通常把元数据分为技术元数据（Technical Metadata）和业务元数据（Business Metadata）。技术元数据是描述关于数据仓库技术细节的数据，这些元数据应用于开发、管理和维护数据仓库；业务元数据从商业和业务的角度描述数据仓库的数据，提供了良好的语义层定义，业务元数据使业务人员能够更好的理解数据仓库分析出来的数据。

技术元数据

技术元数据 - MBA智库百科
技术元数据是指数据仓库的设计和管理人员用于开发和日常管理数据仓库是用的数据。包括：数据源信息；数据转换的描述；数据仓库内对象和数据结构的定义；数据清理和数据更新时用的规则；源数据到目的数据的映射；用户访问权限，数据备份历史记录，数据导入历史记录，信息发布历史记录等。

  • 源系统数据结构定义：导入源系统数据字典，并以系统-表-字段的层级关系展现各级数据关系
  • 主数据ER模型：导入主数据ERwin模型，解析生成模型各层级关系
  • 数据与映射关系定义：定义数据平台各部分，以及源系统间的数据映射关系。
  • 数据影响与血缘分析：依赖所定义的数据平台各部分，图形化展现数据变化影响和数据问题溯源。
  • 数据变化对比分析：针对元数据不同版本进行差异部分识别与比较

业务元数据

业务元数据 - MBA智库百科
业务元数据是数据仓库环境的关键元数据，是用户访问时了解业务数据的途径，内容来源包括多个地方：用例建模(Case Modeling)工具、控制数据库、数据库目录和数据抽 取／转换／加载的工具。另外的业务元数据，例如对象连接关系或数据质量指标，是用元数 据库管理工具直接输入的。

  • 代码定义：定义数据平台公共代码及其源系统代码的关系
  • 报表表头定义：定义数据平台业务报表的构成元素与结构
  • 报表指标定义：定义数据平台报表中指标的计算规则
  • 报表指标钩稽关系分析：定义数据平台业务报表中指标的依赖关系，并进行影响与血缘分析

CWM元模型

在使用元数据的同时，随着数据仓库市场的发展，业界出现许多数据仓库管理和分析的工具，各种工具使用不同的元数据标准来表示和处理，不同系统之间的迁移、数据交换变得困难。于是，我们希望用一种单一的元数据标准，使得各种组织的元数据具有单一的元模型（MetaModel），因此，需要建立一种标准使得不同的数据仓库和商业智能系统之间可以相互交换元数据。
公共仓库元模型CWM(Common Warehouse Metamodel)的主要目的是在异构环境下，帮助不同的数据仓库工具、平台和元数据知识库进行元数据交换。

CWM标准为数据仓库和商业智能（BI）工具之间共享元数据，制定了一整套关于语法和语义的规范，它主要包含以下四个方面的内容：

  • CWM（Metamodel）：描述数据仓库元数据的模型
  • CWM XML：CWM元数据的XML表示
  • CWM DTD：用来验证CWM XML文档
  • CWM IDL：DW/BI共享元数据的应用程序访问接口（API）

CWM1.1的体系结构如图所示，包括五个层次：对象模型层、基础层、资源层、分析层和管理层。

  • 对象模型层（Object Model）：构造和描述其它CWM包中的元模型类。

  • 基础层（Foundation）: 包括表示CWM概念和结构的模型元素，这些模型元素又可被其他CWM包所共享，它由以下六个子包组成：
    1.业务信息（Business Information）包：包括表示模型元素业务信息的类与关联；
    2.数据类型（Data Types）包：包括表示建模者可以用来创建所需数据类型的结构的类与关联；
    3.表达式（Expressions）包：包括表示表达式树的类与关联；
    4.键和索引（Keys and Indexes）包：包括表示键和索引的类与关联；
    5.软件部署（Software Deployment）包：包括软件如何在数据仓库中发布的类与关联；
    6.类型映射（Type Mapping）包：包括表示不同系统之间数据类型映射的类与关联。

  • 资源层（Resource）：用于描述数据资源的包，它包括以下四个子包：
    0.对象（Object）包：包括表示其他类型数据资源的元数据的类与管理；
    1.关系（Relational）包：包括表示关系型数据资源的元数据的类与关联；
    2.记录（Record）包：包括表示记录型数据资源的元数据的类与关联；
    3.多维（Multidimensional）包：包括表示多维数据资源的元数据的类与关联；
    4.XML包：包括表示XML数据资源的元数据的类与关联。

  • 分析层（Analysis）：它由以下五个子包组成：
    1.转换（Transformation）包：包含表示数据抽取和转换工具的元数据的类和关联；
    2.OLAP包：包含表示OLAP工具的元数据的类与关联；
    3.数据挖掘（Data Mining）包：包含表示数据挖掘工具的元数据的类与关联；
    4.信息可视化（Information Visualization）包：包含表示信息可视化工具的元数据的类与关联；
    5.业务术语（Business Nomenclature）包：包括表示分类业务的元数据的类与关联。

  • 管理层（Management）：用于描述数据仓库管理的包，它包括以下两个子包：
    1.仓库过程（Warehouse Process）包：包括表示仓库过程的元数据的类与关联；
    2.仓库操作（Warehouse Operation）包：包括表示仓库操作结果的元数据的类与关联。

CWM作为数据仓库领域的元模型标准，在元数据的集中管理、元数据互操作和元数据交换方面发挥重要作用。CWM是一个国际性、厂商无关、平台无关的规范，提供了一个公共的元模型框架，将数据仓库领域的公共特性纳入元模型中。

主数据管理

主数据（Master Data）指描述核心业务实体的数据，如客户、机构、员工、产品等。这些数据变化相对缓慢并通常在企业内跨业务重复使用。主数据管理适用于管理、协调、监控与企业主要业务实体相关联的主数据的一系列规则、技术、应用、策略和程序。当前大多数企业的主数据跨越几乎所有的组织业务部门，分布在多个不同的系统中。

主数据管理 - MBA智库百科
主数据管理(Master Data Management ，MDM)是指一组约束和方法用来保证一个企业内主题域和系统内相关数据和跨主题域和系统的相关数据的实时性、含义和质量。
